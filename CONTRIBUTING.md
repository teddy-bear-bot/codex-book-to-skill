# Contributing

感谢你关注 Book-to-Skill。

本项目是一个围绕“书籍方法论蒸馏为 Skill”展开的非商业开源项目。我们欢迎对协议、架构、适配器、模板、评估方法和实现代码的贡献。

## 1. 贡献范围

优先欢迎：

- Skill Spec Schema 改进
- 解析与蒸馏流程优化
- 导出适配器
- Playground 评估能力
- 自部署文档
- 示例模板与测试用例

暂不优先：

- 与核心定位无关的大型功能扩展
- 将项目重写为绑定某个单一平台的实现
- 引入会提高版权风险的默认数据集

## 2. 开始前

在开始实现前，请先：

1. 阅读 [README.md](./README.md)
2. 阅读 [book-to-skill-prd.md](./book-to-skill-prd.md)
3. 阅读 [ARCHITECTURE.md](./ARCHITECTURE.md)
4. 检查是否已有相关 issue 或讨论

## 3. 贡献流程

建议流程：

1. 先开 issue 或 discussion 说明目标
2. 对较大改动先提交设计草案
3. 获得维护者确认后再开始实现
4. 提交 PR，并说明动机、范围、风险和验证方式

## 4. Pull Request 要求

PR 应尽量做到：

- 目标单一
- 改动边界清晰
- 包含必要文档
- 包含最小可验证说明
- 不混入无关重构

PR 描述建议包含：

- 改了什么
- 为什么改
- 如何验证
- 是否影响版权边界、Spec 兼容性或导出协议

## 5. 高风险改动

以下改动需要更高 review 门槛：

- Skill Spec Schema 变更
- 导出格式变更
- 公开页数据边界调整
- 与版权或原文展示相关的逻辑变更
- 自部署默认安全策略变更

## 6. 文档与风格

- 优先写清楚设计意图，而不是只给结果
- 公共接口和协议变更需要同步更新文档
- 示例数据不得使用未经授权的书籍文本
- 尽量保持模块边界清晰

## 7. 提交原则

- 小步提交
- 明确说明
- 避免一次 PR 同时改协议、架构和 UI

## 8. 行为准则

贡献者默认受项目行为准则约束。若仓库后续新增 `CODE_OF_CONDUCT.md`，请以该文件为准。

## 9. 安全问题

如果你发现：

- 密钥泄露
- 权限绕过
- 版权边界漏洞
- 可导致用户数据暴露的问题

请不要公开提交 issue。应通过项目维护者指定的安全渠道私下披露。
