# Book-to-Skill PRD

## 1. 产品概述

- 产品名称：Book-to-Skill
- 产品形态：Web 应用
- 项目属性：非商业项目，后续计划开源
- 核心能力：将用户上传的书籍内容蒸馏为平台无关的可安装 Skill
- 核心用户：个人创作者、知识博主
- 核心场景：将书中的方法论封装为内容创作助手，并支持公开发布

一句话定义：

> 把一本书的方法论，编译成可调用、可分享、可迁移到多个 AI 平台的 Skill。

## 2. 目标用户

### 2.1 用户画像

核心用户为个人创作者和知识博主，典型包括：

- 公众号作者
- 小红书/视频号/抖音内容创作者
- 播客主持人
- 课程型创作者
- 高密度阅读商业、写作、营销、心理学、自我成长类书籍的人

### 2.2 用户特征

- 有持续内容产出需求
- 读书频率高，但难以把书中的方法长期用于实际创作
- 已经在使用 AI 写作，但缺少稳定、可复用的方法资产
- 有公开展示个人方法论库的意愿

## 3. 用户问题

当前用户在“读书到应用”的链路中存在以下问题：

1. 读完一本书后，只记得观点，不记得可执行的方法。
2. 笔记和划线很多，但不能直接用于 AI 协作。
3. 很难把一本书稳定封装成反复可调用的 Prompt 或 Skill。
4. 现有读书产品大多停留在摘要、卡片、笔记层，缺少“可执行能力化”。
5. 即使整理出了方法论，也缺少一个可公开展示、可传播的作品形态。

## 4. 产品愿景与定位

### 4.1 产品愿景

帮助创作者把“读过的书”沉淀为“长期可调用的能力资产”。

### 4.2 产品定位

Book-to-Skill 不是：

- 读书笔记工具
- 通用摘要工具
- Prompt 收藏夹

Book-to-Skill 是：

- 书籍方法论蒸馏工具
- Skill 编译与封装平台
- 创作者的 AI 能力资产库

### 4.3 核心价值主张

用户上传一本电子书或 PDF 后，系统自动识别其中的方法论结构，蒸馏为一个平台无关的 Skill Spec，并进一步生成：

- 可编辑的 Skill 定义
- 可测试的 Playground
- 可公开访问的作品页
- 可导出到多个 AI 平台的适配格式

## 5. 产品目标

### 5.1 MVP 目标

验证以下核心假设是否成立：

- 用户真正想要的不是书籍摘要，而是书中方法论的可调用封装
- 创作者愿意将书籍转化为 Skill 并在内容创作中反复使用
- 用户愿意公开发布自己生成的 Skill

### 5.2 用户目标

- 把一本书快速变成一个可用的创作助手
- 在真实写作和选题场景中反复调用
- 将自己的方法论库沉淀并公开展示

### 5.3 项目目标

- 让用户在 10 分钟内完成从上传书籍到 Skill 初稿生成
- 提高 Skill 在 7 日内的重复调用率
- 提高 Skill 的公开发布率和分享率

## 6. 关键假设

1. 书的长期价值主要体现在方法论结构，而不是原文摘要。
2. 创作者更愿意持续使用和维护“可复用能力资产”，而不是一次性摘要结果。
3. 平台无关的 Skill Spec 适合作为中间层，便于后续多平台适配。
4. 只要展示页足够作品化，用户愿意公开分享自己的 Skill。
5. 开源形态有助于增强透明度、可迁移性和社区协作。

## 7. 非目标

MVP 阶段不做以下内容：

- 通用知识库管理
- 团队协作与权限系统
- 交易、付费分发和计费体系
- Kindle、微信读书等外部平台直连
- 多平台原生一键安装
- 社区推荐流和复杂社交互动

## 8. 核心场景

### 8.1 主场景：内容创作

用户上传一本关于写作、表达、营销、定位的书。系统识别其中的方法论，生成一个适合内容创作的 Skill。用户在写公众号、播客提纲、短视频脚本时调用该 Skill，让 AI 按书中的逻辑协助输出。

### 8.2 次场景：选题与结构设计

用户上传一本关于叙事、品牌、增长或定位的书，将其转化为 Skill，用于生成选题角度、论证结构、脚本框架。

### 8.3 次场景：公开作品展示

用户把自己读过的重要书逐步 Skill 化，形成公开方法论作品页，作为个人品牌的一部分。

## 9. 产品方案

### 9.1 整体流程

产品主流程分为 5 个阶段：

1. 上传书籍
2. 解析内容
3. 蒸馏方法论
4. 生成并编辑 Skill
5. 测试、发布、分享

### 9.2 输入边界

MVP 首版支持：

- 用户上传 PDF
- 用户上传 EPUB
- 可解析文本型电子书

首版不重点支持：

- 用户粘贴笔记
- 导入 Kindle 划线
- 导入微信读书笔记
- 扫描版图书的高质量 OCR 优化

## 10. 核心功能

### 10.1 书籍上传与解析

功能目标：让用户可以顺利将书籍导入系统，并获得结构化文本。

功能点：

- 上传 PDF / EPUB
- 自动文本抽取
- 章节目录识别
- 基础噪音过滤
- 解析失败提示

关键要求：

- 清晰反馈上传进度、解析进度、失败原因
- 对扫描件或低质量文档提示可解析性风险
- 用户可在解析后预览目录与章节结构

### 10.2 方法论蒸馏引擎

功能目标：从整本书中抽取“可指导行动”的结构，而不是只生成摘要。

重点识别内容：

- 核心命题
- 一级框架
- 方法步骤
- 判断标准
- 适用边界
- 反模式/误区
- 典型案例
- 可迁移场景

产出要求：

- 保持原书方法论特色
- 尽量避免泛化为空洞原则
- 对每条结论保留可追溯依据

### 10.3 Skill Spec 生成器

功能目标：将蒸馏结果转化为平台无关的通用 Skill 定义。

建议字段：

- `title`
- `summary`
- `source_book`
- `target_user`
- `goal`
- `applicable_scenarios`
- `required_inputs`
- `core_principles`
- `workflow`
- `decision_rules`
- `constraints`
- `anti_patterns`
- `output_format`
- `example_tasks`
- `evaluation_checklist`
- `platform_adapters`

### 10.4 Skill 编辑器

功能目标：让用户在 AI 初稿基础上做最小成本修正，而不是从零创作。

可编辑内容：

- Skill 名称
- Skill 简介
- 使用场景
- 方法步骤
- 输出风格
- 示例任务
- 不适用边界

设计原则：

- 默认提供高质量初稿
- 编辑过程尽量结构化
- 支持版本保存

### 10.5 Playground

功能目标：让用户在真实创作任务中验证 Skill 是否有用。

核心能力：

- 输入具体创作任务
- 调用所选 Skill
- 展示生成结果
- 展示调用了哪些原则或步骤
- 支持用户继续修改 Skill 并重试

示例任务：

- 帮我输出一篇公众号文章提纲
- 基于这本书的方法帮我设计 5 个选题
- 用这套框架重写一段视频脚本开头

### 10.6 发布页

功能目标：让 Skill 成为可以展示和传播的作品。

页面内容：

- Skill 名称
- 来源书籍
- Skill 简介
- 适用场景
- 示例输出
- 作者信息
- 使用入口或导出入口
- 分享链接

要求：

- 页面风格更像作品页，而不是后台详情页
- 公开信息以“方法论封装结果”为主，而不是原书复刻

### 10.7 导出适配器

功能目标：支持 Skill 向不同 AI 平台迁移。

MVP 输出：

- Markdown Skill Spec
- JSON 格式 Skill 定义
- 通用 Prompt 包

后续扩展：

- 面向具体平台的导出模板
- 一键安装适配

## 11. 用户流程

### 11.1 从书到 Skill

1. 用户注册并进入工作台
2. 上传一本 PDF / EPUB
3. 系统完成解析并展示目录预览
4. 系统蒸馏书籍方法论
5. 系统生成 Skill 初稿
6. 用户进入编辑器调整 Skill
7. 用户在 Playground 中测试效果
8. 用户选择公开发布
9. 系统生成可分享作品页

### 11.2 从 Skill 到创作

1. 用户进入自己的 Skill 库
2. 选择一个 Skill
3. 输入当前创作任务
4. 查看生成结果和方法依据
5. 调整 Skill 或导出使用

## 12. MVP 范围

### 12.1 必须包含

- 用户注册登录
- PDF / EPUB 上传
- 基础解析与章节识别
- 方法论蒸馏
- Skill Spec 自动生成
- Skill 编辑器
- Playground 测试
- 公开发布页
- 链接分享
- 通用格式导出

### 12.2 可延后

- 用户主页
- 评论、点赞、收藏
- 外部阅读平台接入
- 多人协作
- 批量处理多本书
- 社区模板库

## 13. 差异化

相较于读书摘要工具，Book-to-Skill 的核心差异在于：

1. 输出对象不是摘要，而是可执行 Skill。
2. 使用目标不是阅读，而是调用。
3. 价值不止在于压缩信息，而在于提炼方法并反复复用。
4. Skill 可以成为创作者对外展示的作品，而不是私密笔记。
5. 采用“通用 Skill Spec + 多平台适配”的中间层架构，避免过早绑定单一平台。

## 14. 风险与约束

### 14.1 版权风险

这是 MVP 的最高优先级风险。

处理原则：

- 不公开展示整本书内容
- 不允许大段原文复刻式输出
- 公开页以蒸馏后方法论为主
- 用户需确认拥有合法使用权
- 平台保留下架和审核能力

### 14.2 质量风险

主要风险：

- AI 误解作者观点
- 将案例误识别为原则
- Skill 过度抽象，失去原书独特性

应对方式：

- 展示蒸馏依据
- 支持用户删除和修订原则
- 引入质量评分与人工校正机制

### 14.3 平台适配风险

问题：

- 不同 AI 平台对 Skill 机制支持不同
- 原生安装能力和格式差异大

应对原则：

- 首版坚持平台无关 Spec 优先
- 平台适配器作为第二层能力逐步扩展

## 15. 数据指标

### 15.1 北极星指标

- 7 日内被重复调用的 Skill 数量

### 15.2 核心项目指标

- 上传成功率
- 解析完成率
- Skill 生成完成率
- Playground 使用率
- 7 日内二次调用率
- 公开发布率
- 分享率
- 导出率

### 15.3 质量指标

- 用户对生成 Skill 的编辑幅度
- 用户满意度评分

- 发布页访问到调用转化率

## 16. 开源与发布策略

### 16.1 项目原则

- 不做商业化，不设计免费版和 Pro 版分层
- 核心能力优先以统一版本提供，不靠功能阉割区分用户
- 优先保证 Skill Spec、导出格式和核心流程的开放性
- 避免将项目设计为依赖封闭平台能力的黑盒工具

### 16.2 License 建议

建议默认采用分层授权：

- 代码仓库使用 `MIT` 或 `Apache-2.0`
- Skill Spec、Schema、示例配置使用宽松开放协议
- 文档内容使用 `CC BY 4.0`

推荐默认组合：

- 代码层：`Apache-2.0`
- 文档层：`CC BY 4.0`

推荐原因：

- 对开发者和集成方更清晰
- 兼顾开放性与后续生态协作

### 16.3 开源边界

建议开源的内容：

- Skill Spec Schema
- 书籍解析流水线
- 方法论蒸馏框架
- 导出适配层
- Web 前后端核心代码
- 自部署脚本与部署文档

建议不进入默认仓库分发的内容：

- 用户上传的原始书籍文件
- 受版权约束的原文片段缓存
- 第三方模型密钥与私有配置
- 含个人隐私或使用记录的数据集

边界原则：

- 开源的是工具链、协议与工作流，不是受版权保护的原始内容
- 示例数据优先使用公共领域文本、自建样本或人工构造内容
- 仓库中不默认附带任何未经授权的书籍文本或派生数据

### 16.4 社区治理

建议在开源初期就建立最小治理框架：

- `README`：项目定位、目标与边界
- `CONTRIBUTING`：贡献流程、代码规范、提交流程
- `CODE_OF_CONDUCT`：社区行为准则
- `SECURITY`：漏洞披露方式
- `GOVERNANCE`：维护者职责、评审与决策机制

治理原则：

- 初期保持 maintainer-driven，避免治理结构过重
- Schema 变更、版权边界、导出协议相关改动提高 review 门槛
- 优先鼓励社区贡献模板、适配器和评估用例

### 16.5 自部署架构

自部署建议作为开源路线的核心组成部分。

建议分层：

- Web 前端：上传、编辑、Playground、发布页
- API 服务：编排解析、蒸馏、导出、权限逻辑
- Worker 服务：执行长任务，如抽取、分段、蒸馏、导出
- 数据层：存储用户、Skill、任务状态、元数据
- 对象存储：存储上传文件、导出文件、发布素材

建议的最小部署单元：

- 1 个 Web/API 服务
- 1 个异步 Worker
- 1 个数据库
- 1 个对象存储或本地文件存储适配层

设计要求：

- 模型供应商可替换，不与单一厂商耦合
- 文件存储可替换，兼容本地存储和 S3 兼容存储
- 任务队列可替换，避免把调度逻辑写死
- 平台适配器模块化，便于增删和社区扩展

### 16.6 开源方向

- 后续以开源项目方式发布核心代码
- 优先开放 Skill Spec、解析流程和导出适配层
- 鼓励社区贡献新的平台适配器、模板和评估机制
- 支持开发者自部署和二次扩展

### 16.7 发布策略

- Skill 合集
- 创作者主页
- 社区模板库
- 自部署文档
- 开发者集成文档
