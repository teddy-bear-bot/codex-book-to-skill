# Governance

## 1. 目标

本治理文档用于说明 Book-to-Skill 在开源初期的维护方式、决策机制和责任边界。

项目当前采用 maintainer-driven 模式，先保证方向集中、边界清晰和演进稳定。

## 2. 治理原则

- 优先保护项目定位的一致性
- 优先保护版权边界和用户数据边界
- 优先保护 Skill Spec 的稳定性
- 鼓励扩展，但反对无边界膨胀

## 3. 角色

### 3.1 Maintainers

职责：

- 维护项目方向
- 审核高风险改动
- 决定 Spec 演进节奏
- 决定发布策略
- 处理冲突与仲裁

### 3.2 Contributors

职责：

- 提交 issue、文档、代码和适配器
- 遵守贡献流程
- 避免引入版权和安全风险

### 3.3 Community Contributors

适合贡献：

- 模板
- 适配器
- 文档修订
- 评估集和测试样例

## 4. 决策分层

### 4.1 可直接评审合并的改动

- 文档修正
- 小型 bug 修复
- 非破坏性的测试补充
- 新增不影响核心协议的模板

### 4.2 需要维护者明确批准的改动

- Skill Spec Schema 变更
- 导出协议变更
- 架构层级调整
- 自部署默认策略变更
- 与版权边界相关的逻辑改动

## 5. 决策方式

当前默认采用：

- 讨论优先
- 维护者做最终裁决

如果后续社区规模扩大，再考虑引入更正式的 RFC 流程。

## 6. 版本与兼容性

关于 Skill Spec 和适配接口：

- 尽量保持向后兼容
- 破坏性调整应提前公告
- 重大变更建议先走设计讨论

## 7. 版权与数据治理

这是项目的红线领域。

基本原则：

- 不将受版权保护的原始书籍文本作为开源资产分发
- 不默认公开用户上传内容
- 不接受来路不明的数据集进入默认仓库

## 8. 争议处理

若出现以下争议：

- 项目方向分歧
- 协议设计争议
- 版权边界争议
- 安全处理争议

由维护者先组织讨论，必要时给出书面结论并记录。

## 9. 后续演进

当项目进入更成熟阶段，可逐步补充：

- RFC 流程
- Maintainer 进入与退出规则
- 发布节奏说明
- 安全响应 SLA
- 社区工作组机制
