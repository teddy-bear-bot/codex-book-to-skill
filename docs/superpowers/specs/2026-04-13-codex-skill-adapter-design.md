# Codex Skill Adapter Design

## Background

`book-to-skill` can already generate portable `.tar.gz` Skill packages, install them into its own registry, and open a persistent host shell with `book-to-skill chat`.

The current chat host still uses a placeholder runtime for ordinary user text after a Skill is selected. The product direction is different: `book-to-skill` should not become its own model runtime. It should install, manage, and launch Skills for established agent hosts such as Codex, Claude Code, and Gemini.

This phase implements the first host adapter: Codex.

## Goal

When a user selects a packaged Skill in `book-to-skill chat`, `book-to-skill` should convert that package into a Codex-native Skill, install it into the user-level Codex Skills directory, and then launch Codex so Codex handles the real interaction.

The first target flow is:

```bash
book-to-skill chat ./pyramid-writing-skill.tar.gz
```

Then inside the host:

```text
/skill
1
```

After selection, `book-to-skill` should:

1. sync the selected Skill to `~/.codex/skills/<skill-id>`
2. report that the Codex Skill was installed
3. launch the local `codex` command if available
4. let the user continue naturally inside Codex
5. otherwise print a manual fallback message

## Non-Goals

This phase does not implement:

- Claude Code adapter support
- Gemini adapter support
- model-provider calls inside `book-to-skill`
- multiple active Skills in one launched Codex session
- full-screen TUI behavior
- automatic Skill updates from GitHub
- complex version-selection UI
- temporary-only Skill injection

## Product Model

`book-to-skill` is the package and host manager.

Codex is the execution host.

That means:

- `book-to-skill` installs and selects Skills
- `book-to-skill` converts a selected Skill into Codex's Skill shape
- Codex reads and applies the generated `SKILL.md`
- ordinary model conversation happens in Codex, not in `book-to-skill`

## Codex Install Location

The first version installs generated Codex Skills directly into the user-level Codex directory:

```text
~/.codex/skills/<skill-id>
```

Example:

```text
~/.codex/skills/pyramid-writing-skill
```

This matches the user's preferred behavior: once converted, the Skill is globally available to future Codex sessions for that user.

## Generated Codex Skill Shape

Each generated Codex Skill should contain:

```text
<skill-id>/
├── SKILL.md
└── references/
    ├── system.md
    ├── inputs.schema.json
    └── book-to-skill.json
```

### `SKILL.md`

`SKILL.md` is the Codex-native entrypoint. It should be concise and should not paste the full book-derived system prompt inline.

It should contain YAML frontmatter:

```yaml
---
name: pyramid-writing-skill
description: Use when the user wants help applying the packaged Pyramid Writing Skill from book-to-skill.
---
```

Its body should instruct Codex to:

- use this Skill when the user asks for the packaged capability
- read `references/system.md` for the full method and behavioral instructions
- read `references/inputs.schema.json` when it needs the expected input fields
- avoid reproducing copyrighted source text
- follow the original constraints from the packaged Skill

### `references/system.md`

This file is copied from the installed book-to-skill package's configured `entrypoints.system_prompt_file`, usually `system.md`.

### `references/inputs.schema.json`

This file is copied from the installed book-to-skill package's configured `entrypoints.inputs_schema_file`, usually `inputs.schema.json`.

### `references/book-to-skill.json`

This provenance file records that the Codex Skill was generated by `book-to-skill`, for overwrite safety and debugging.

Suggested shape:

```json
{
  "generated_by": "book-to-skill",
  "generated_at": "2026-04-13T00:00:00.000Z",
  "source_skill": {
    "id": "pyramid-writing-skill",
    "name": "金字塔结构写作 Skill",
    "version": "1.0.0",
    "installed_path": "/.../pyramid-writing-skill@1.0.0"
  }
}
```

## Overwrite Policy

If `~/.codex/skills/<skill-id>` does not exist, create it.

If it exists and contains `references/book-to-skill.json` with:

```json
{
  "generated_by": "book-to-skill"
}
```

then overwrite it.

If it exists but does not contain that provenance marker, fail safely and print a clear error. This prevents accidentally overwriting a user's hand-written Codex Skill.

## Chat Host Behavior

When the user selects a Skill via `/skill`, the current placeholder runtime should be replaced by Codex handoff behavior.

Expected first-version behavior:

1. set the selected Skill as active
2. sync the selected Skill into `~/.codex/skills/<skill-id>`
3. print a line such as:

```text
Installed Codex skill: ~/.codex/skills/pyramid-writing-skill
```

4. attempt to launch Codex by spawning `codex`
5. if `codex` starts, let it inherit the terminal so the user is inside Codex
6. in that Codex session, the user continues by typing a normal request; Codex should discover the installed Skill through its normal Skill mechanism
7. the first version does not require `book-to-skill` to pre-select or explicitly inject the Skill at Codex launch time
8. when Codex exits, return control to the terminal; the first version may then end the `book-to-skill chat` process
7. if `codex` is not found, print a fallback message:

```text
Codex skill installed at ~/.codex/skills/pyramid-writing-skill.
Open Codex manually, then ask for help using pyramid-writing-skill.
```

## Launch Command

The first version should prefer a conservative launch:

```bash
codex
```

If later we confirm a stable Codex CLI flag for opening with a specific Skill or initial prompt, that can be added as a separate enhancement. The first version should rely on Codex's normal Skill discovery from `~/.codex/skills`.

## Activation Behavior in Codex

The first version does not attempt to force-select the generated Skill after launching Codex.

Instead, the expected flow is:

1. `book-to-skill` installs the generated Skill into `~/.codex/skills/<skill-id>`
2. `book-to-skill` launches `codex`
3. the user types a normal request in Codex, such as:

```text
Help me use pyramid-writing-skill to draft an outline about remote work for startup managers.
```

4. Codex decides whether to trigger the installed Skill using its standard Skill selection logic

This behavior is intentional for v1 because the install path is stable, while explicit Skill activation flags are not yet confirmed.

## Error Handling

### Missing Skill files

If the installed book-to-skill package lacks the configured `system.md` or `inputs.schema.json`, syncing should fail with a clear error and the host should stay alive.

### Existing user-created Codex Skill

If the target Codex Skill directory already exists and was not generated by `book-to-skill`, syncing should fail safely and explain that it will not overwrite user content.

### Missing Codex CLI

If `codex` is not available on `PATH`, this should not undo the install. The user should get a manual fallback message that includes:

- where the generated Codex Skill was installed
- that they can open Codex manually
- that they should ask for the packaged Skill by name in natural language

### Spawn failure

If `codex` fails to launch for another reason, print the error and keep the shell from crashing with an unclear stack trace.

## Testing Strategy

Use `node:test` with no external network dependency.

Add tests for:

- converting an installed Skill into a Codex Skill directory
- generated `SKILL.md` contains valid frontmatter and references the copied files
- references are copied from the original package entrypoints
- provenance file is written
- generated Codex Skills can be overwritten safely
- user-created existing Codex Skill directories are not overwritten
- `/skill` selection triggers Codex sync
- missing `codex` command produces fallback output without failing the whole host

Use a temporary Codex Skills root in tests instead of writing to `~/.codex/skills`.

## Open Questions Deferred

- Exact Codex CLI flags for launching with a specific active Skill
- Whether to return to the `book-to-skill` host after Codex exits
- Whether to support multi-host installs in one command, such as `--host codex|claude|gemini`
- Whether to add a separate `book-to-skill codex install <skill>` command
